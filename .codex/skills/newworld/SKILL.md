# NewWorld

## 作用范围检查

使用本 skill 前必须先确认任务属于 `C:\NewWorld` 项目。

只有满足以下任一条件时才使用：

- 当前工作目录位于 `C:\NewWorld`。
- 用户明确提到 NewWorld 项目。
- 用户要求修改的文件位于 `C:\NewWorld`。

如果任务不属于 `C:\NewWorld`，立即停止使用本 skill，改按普通上下文处理。

## 项目概述

NewWorld 是一个面向小说、剧本、游戏设定和长篇叙事资料的世界构建与剧情推演工具。用户输入剧情文本、人物设定、世界观设定等资料后，系统抽取重要节点和关系，构建可编辑的关系图谱与世界观档案，并基于时间线推演剧情发展走向。

这里的节点不一定是人，也可以是组织、地点、国家、种族、物品、事件、规则、能力、信仰、秘密或任何会影响剧情的关键设定。

项目根目录：`C:\NewWorld`

做架构、模块、schema、路线图或产品范围决策时，读取 `docs/architecture.md` 和 `references/project-architecture.md`。

## 必要需求优先级

开发时优先满足这些需求：

1. 抽取小说剧情、人物设定、世界观设定等文字资料，构建重要节点关系图谱。
2. 点代表人物或重要设定节点，线代表关系。
3. 根据抽取资料构建世界观设定。
4. 根据关系图谱、世界观设定和时间线推演剧情发展走向。
5. 支持用户手动添加设定人物或其他重要节点。
6. 支持通过时间轴查看不同时间点的人物信息、节点状态与剧情发展。
7. 提供清晰的时间发展时间线。

## 开发流程

1. 修改前先检查仓库状态和现有文件。
2. 优先遵循 `docs/architecture.md` 和 `references/project-architecture.md`。
3. 早期实现保持窄、可运行、可继续演进。
4. 前端第一屏做真实创作工作台，不做营销落地页。
5. 图谱、世界观设定、时间线是同等重要的一等数据。
6. 模型输出入库前必须做 schema 校验。
7. 用户手动添加或编辑的设定优先级高于模型建议。
8. 用户要求提交时，按清晰里程碑提交。

## 产品词汇

优先使用简单词汇：

- 项目：一个故事或世界总容器
- 来源：输入资料
- 节点：人物或重要设定对象
- 地图：节点和关系图谱
- 设定库：世界观设定条目集合
- 时间线：剧情和历史事件的发展线
- 状态：某个节点在某个时间点的信息
- 推演：剧情发展走向预测
- 报告：推演总结与解释

避免过度学术化。界面文案要清楚，有一点创作工具的探索感，但不要牺牲专业性。

## 推荐架构

初始技术栈：

- 前端：React + TypeScript + Vite
- 后端：Python FastAPI
- 存储：MVP 使用 SQLite，后续 PostgreSQL
- 可视化：React Flow 优先
- 时间线：先自研轻量组件
- LLM 接入：模型适配器接口
- 测试：默认使用 mock LLM provider

核心后端服务：

- 资料抽取服务
- 图谱服务
- 世界观设定服务
- 时间线服务
- 剧情推演服务
- 报告服务
- 模型适配器
- 持久化仓储层

## 实现原则

- 节点不等于人物，必须支持所有重要设定对象。
- 图谱、世界观设定和时间线要互相关联。
- 用户编辑状态和模型生成建议分开存储。
- 保存来源引用，方便追溯每个节点、关系、设定和事件。
- 时间线要能驱动不同时间点的节点状态和关系变化。
- 剧情推演要解释因果，不只生成一段故事文本。
- 测试中使用确定性的 mock provider。

## MVP 主流程

围绕这个流程推进：

1. 创建项目。
2. 添加剧情、人物设定或世界观设定文本作为来源。
3. 抽取重要节点和关系，形成地图。
4. 抽取世界观设定，形成设定库。
5. 抽取或手动创建时间线事件。
6. 手动补充节点、关系或设定。
7. 选择时间点，查看节点状态与剧情发展。
8. 基于图谱、设定库和时间线推演剧情走向。
9. 生成带因果解释和分支建议的报告。
